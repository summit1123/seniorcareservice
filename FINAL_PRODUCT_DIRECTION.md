# 최종 기획 방향

## 1. 한 줄 정의

이 프로젝트는 기존 마일리지 할인특약에 단순 추가 혜택을 붙이는 것이 아니라, **연환산 주행거리, 생활권 내 안정운전, 생활권 밖 안전운전, 생활권 밖 위험변화**를 함께 반영해 마일리지 판단 기준을 고도화하는 **시니어 특화 통합 마일리지 산정 모델**이다.

핵심 문제의식은 아래와 같다.

```text
기존 마일리지는 같은 저주행 고객을 동일하게 본다.
하지만 같은 4,000km를 운전한 시니어라도
생활권 안에서 반복적으로 안정 운전한 고객과
최근 생활권 밖 야간/급감속/과속이 늘어난 고객은 다르게 봐야 한다.
```

따라서 우리 제안은 "시니어라서 더 깎아주는 상품"이 아니다.

```text
기존 마일리지가 놓치는 저주행 고객 내부의 위험 차이를
생활권 AI와 위험변화 지표로 재분류하는 상품 설계안이다.
```

## 2. 최종 플로우

전체 흐름은 아래 순서로 고정한다.

```text
1. 페르소나 기반 배치 시나리오 시뮬레이터
   -> 기존 마일리지와 제안 마일리지 산식의 조건 검증
   -> 생활권 기준, 위험변화 기준, 케어 기준, 가중치 후보 산출

2. 기준 확정
   -> 제안 특약이 어떤 기준으로 우대/기본/예방 케어를 나눌지 결정

3. 실제 고객 주행 데이터 분석
   -> DBSCAN으로 개인 생활권 중심 생성
   -> P90 버퍼로 생활권 범위 보정
   -> 생활권 안/밖 위험운전 피처 계산
   -> 위험도/위험변화 점수 산출

4. 통합 마일리지 산정
   -> 연환산 주행거리
   -> In-Zone Safe Driving Score
   -> Out-Zone Safe Driving Score
   -> Out-Zone Pattern Change Risk
   -> Senior Safe Mileage Score 산출

5. 상품 판정
   -> 우대 / 기본 / 예방 케어 분류

6. XAI + LLM 리포트
   -> XAI가 판정 이유를 추출
   -> LLM이 보험사 직원용 설명문으로 변환
```

역할을 나누면 다음과 같다.

| 구성 | 역할 |
|---|---|
| 페르소나 기반 배치 시나리오 시뮬레이터 | 상품 산식과 기준값을 출시 전에 검증 |
| 생활권 AI 분석 엔진 | 실제 고객 주행 데이터에서 생활권과 위험변화 계산 |
| 통합 마일리지 산정 모델 | 거리 중심 마일리지 판단을 종합 지표 기반으로 고도화 |
| XAI + LLM 리포트 | 보험사 직원이 판정 이유를 납득할 수 있게 설명 |

## 3. 기존 마일리지와 제안 모델의 차이

기존 마일리지 할인특약은 연환산 주행거리로 할인율을 계산한다.

```text
연환산 주행거리
= (최종 주행km - 최초 주행km)
  / (최종사진 제출일자 - 최초사진 제출일자)
  * 365일
```

기존 방식은 명확하고 운영이 쉽지만, 저주행 고객 내부의 위험 차이를 구분하지 못한다.

| 구분 | 기존 마일리지 | 제안 통합 마일리지 |
|---|---|---|
| 핵심 기준 | 연환산 주행거리 | 주행거리 + 생활권 내 안정운전 + 생활권 밖 안전운전 + 생활권 밖 위험변화 |
| 같은 저주행 고객 구분 | 어려움 | 가능 |
| 생활권 안 위험행동 | 반영 불가 | 반영 |
| 생활권 밖 안정주행 | 반영 불가 | 불필요하게 불리하지 않게 보정 |
| 최근 위험변화 | 반영 불가 | 예방 케어 신호로 반영 |
| 최종 목적 | 적게 탄 고객 할인 | 저주행 고객 내부의 안정형/위험변화형 재분류 |

중요한 표현은 아래로 고정한다.

```text
기존 마일리지 할인 + 추가 리워드가 아니라,
기존 거리 중심 마일리지 산식을 종합 안심주행 지표 기반으로 고도화한다.
```

## 4. 최종 고정 지표

최종 모델의 핵심 지표는 아래 4개로 고정한다.

| 지표 | 의미 | 역할 |
|---|---|---|
| `Mileage Score` | 연환산 주행거리가 낮을수록 높은 점수 | 기존 마일리지 기준 반영 |
| `In-Zone Safe Driving Score` | 생활권 안/버퍼 안에서 반복적으로 안정 운전했는지 | 시니어 생활권 안정성 핵심 |
| `Out-Zone Safe Driving Score` | 생활권 밖에서도 과속/급감속/야간주행이 낮은지 | 생활권 밖 주행을 무조건 위험으로 보지 않기 위한 보정 |
| `Out-Zone Pattern Change Risk` | 최근 생활권 밖 주행과 위험행동이 평소보다 증가했는지 | 예방 케어 트리거 |

초기 산식은 아래 형태로 둔다.

```text
Senior Safe Mileage Score
= w1 * Mileage Score
+ w2 * In-Zone Safe Driving Score
+ w3 * Out-Zone Safe Driving Score
- w4 * Out-Zone Pattern Change Risk
```

초기 후보 가중치는 아래처럼 둘 수 있다.

| 가중치 | 후보값 | 의미 |
|---|---:|---|
| `w1` | 0.35 | 기존 주행거리 기준 |
| `w2` | 0.35 | 생활권 내 안정운전 |
| `w3` | 0.15 | 생활권 밖 안전운전 |
| `w4` | 0.15 | 생활권 밖 위험변화 감점 |

단, 이 숫자는 최종 확정값이 아니다.

```text
가중치와 등급 기준은 페르소나 기반 배치 시나리오 시뮬레이터로 비교한 뒤 확정한다.
```

## 5. 생활권 안/밖 해석 원칙

이 모델은 생활권 안/밖을 단순 위험 라벨로 사용하지 않는다.

```text
생활권 안 = 무조건 안전 아님
생활권 밖 = 무조건 위험 아님
```

정확한 해석은 아래와 같다.

| 상황 | 해석 |
|---|---|
| 생활권 안에서 반복적이고 안정적으로 운전 | 우량 신호 |
| 생활권 안에서 과속/급감속/급가속 증가 | 감점 신호 |
| 생활권 밖이지만 과속/급감속/야간주행이 낮음 | 중립 또는 일부 안정 신호 |
| 생활권 밖 주행이 최근 증가하고 위험행동도 함께 증가 | 예방 케어 신호 |

따라서 핵심은 "생활권 밖이라서 위험"이 아니라 아래 조합이다.

```text
생활권 밖 주행 증가
+ 야간주행 증가
+ 급감속/과속 등 위험행동 증가
+ 평소 대비 패턴 변화
= 예방 케어 검토 신호
```

## 6. 페르소나 기반 배치 시나리오 시뮬레이터

시뮬레이터는 메인 서비스가 아니다.

정확한 역할은 아래와 같다.

```text
상품 출시 전에 다양한 페르소나와 엣지케이스 주행 데이터를 batch로 생성하고,
같은 데이터에 기존 마일리지 산식과 제안 통합 마일리지 산식을 적용해
어떤 기준값이 가장 납득 가능한지 검증하는 내부 상품 설계 도구다.
```

이 시뮬레이터는 실제 사고율이나 손해율을 증명하지 않는다.

검증하는 것은 아래다.

```text
기존 마일리지가 동일하게 처리하는 저주행 고객군 안에서
제안 모델이 안정형 고객과 위험변화형 고객을 더 잘 구분하는가
```

### 6.1 페르소나 구성

초기 구현은 30명, 6개 유형, 각 5명으로 둔다.

| 유형 | 검증 목적 |
|---|---|
| 안정 저주행형 | 기존/제안 모두 우량으로 나와야 하는 기본 우량군 |
| 보통 생활권형 | 일반적인 기본 유지군 |
| 저주행 위험변화형 | 기존은 저주행 할인 대상이나 제안 모델은 케어 신호를 잡아야 하는 핵심 케이스 |
| 장거리 안정형 | 많이 타지만 위험행동은 낮은 케이스 |
| 생활권 안 위험행동형 | 생활권 안이어도 과속/급감속이 있으면 감점되는지 확인 |
| 생활권 밖 안정형 | 생활권 밖 주행 자체를 과도하게 불리하게 보지 않는지 확인 |

### 6.2 관측 기간

90일은 실제 수집 기간이 아니라 시뮬레이션상의 가상 관측 기간이다.

```text
baseline 60일: 평소 생활권 생성
recent 30일: 최근 위험변화 탐지
```

### 6.3 생성할 주행 데이터

시뮬레이터는 OpenAI Batch API로 숫자 데이터를 만들지 않는다.

좌표, 거리, 시간, 위험행동 값의 일관성을 위해 로컬 Python 시뮬레이터로 생성한다.

생성 필드는 아래를 기본으로 한다.

| 필드 | 의미 |
|---|---|
| `driver_id` | 운전자 ID |
| `trip_id` | 주행 ID |
| `trip_date` | 주행일 |
| `start_time` | 출발 시각 |
| `end_time` | 도착 시각 |
| `start_lat`, `start_lon` | 출발 좌표 |
| `end_lat`, `end_lon` | 도착 좌표 |
| `distance_km` | 주행거리 |
| `duration_min` | 주행시간 |
| `night_flag` | 야간 여부 |
| `speeding_count` | 과속 이벤트 |
| `harsh_brake_count` | 급감속 이벤트 |
| `harsh_accel_count` | 급가속 이벤트 |
| `persona_type` | 페르소나 유형 |

## 7. A/B 상품룰 검증

같은 주행 데이터를 두 산식에 넣어 비교한다.

### A안: 기존 마일리지

```text
연환산 주행거리 -> 기존 마일리지 할인 구간
```

### B안: 제안 통합 마일리지

```text
연환산 주행거리
+ In-Zone Safe Driving Score
+ Out-Zone Safe Driving Score
- Out-Zone Pattern Change Risk
-> Senior Safe Mileage Score
-> 우대 / 기본 / 예방 케어
```

여기서 "우대"는 무조건적인 추가 할인이 아니다.

```text
통합 점수 기반으로 기존 거리 중심 판단보다 더 적합한 마일리지 등급을 산정한다.
```

## 8. 시뮬레이션에서 조절할 기준

너무 많은 변수를 동시에 조절하면 설명력이 떨어지므로, 초기에는 아래 항목만 조절한다.

| 기준 | 의미 | 후보 |
|---|---|---|
| `w_mileage` | 주행거리 가중치 | 0.30, 0.35, 0.40 |
| `w_in_zone` | 생활권 내 안정운전 가중치 | 0.30, 0.35, 0.40 |
| `w_out_zone_safe` | 생활권 밖 안전운전 가중치 | 0.10, 0.15, 0.20 |
| `w_out_zone_change` | 생활권 밖 위험변화 감점 가중치 | 0.10, 0.15, 0.20 |
| `care_threshold` | 예방 케어 기준 | 위험변화 상위 20%, 15%, 10% |
| `tier_threshold` | 등급 기준 | S/A/B/C 점수 구간 |

최종 기준은 아래 조건을 만족하는 조합으로 선택한다.

```text
1. 저주행 위험변화 고객을 기존 마일리지보다 잘 포착한다.
2. 안정 저주행 고객을 과도하게 케어 대상으로 보내지 않는다.
3. 생활권 밖 안정형 고객을 무조건 불리하게 보지 않는다.
4. 생활권 안 위험행동형 고객을 놓치지 않는다.
5. 케어 대상 비율이 보험사 운영상 감당 가능하다.
6. 우대 대상 비율과 비용이 과도하지 않다.
```

## 9. 시뮬레이션 산출 지표

시뮬레이션 결과는 사고율이나 실제 손해율이 아니라, 상품 기준의 타당성을 설명하는 기대값으로 제시한다.

| 지표 | 의미 |
|---|---|
| `low_mileage_high_risk_capture` | 기존 마일리지가 할인하는 저주행 위험변화 고객을 제안 모델이 포착한 비율 |
| `stable_driver_capture_rate` | 안정 저주행 고객을 안정형으로 분류한 비율 |
| `care_target_rate` | 예방 케어 대상 비율 |
| `priority_or_favorable_rate` | 우대 또는 유리한 등급 대상 비율 |
| `risk_exposure_index` | 과속, 급감속, 야간, 생활권 밖 위험변화를 종합한 위험노출 지수 |
| `misclassification_check` | 생활권 안 위험형/생활권 밖 안정형을 제대로 구분했는지 |
| `insurer_efficiency_score` | 우대 비용 대비 위험군 분리 효율 |

## 10. 실제 고객 데이터 운영 플로우

시뮬레이터로 기준값을 정한 뒤, 실제 운영에서는 고객 주행 데이터에 아래 AI 분석을 적용한다.

```text
고객 주행 데이터
-> baseline/recent 기간 분리
-> DBSCAN 기반 생활권 중심 생성
-> P90 버퍼 계산
-> core / buffer / outer zone 구분
-> 생활권 안/밖 위험운전 피처 계산
-> 4개 최종 지표 산출
-> Senior Safe Mileage Score 계산
-> 우대 / 기본 / 예방 케어 판정
-> XAI reason code 생성
-> LLM 보험사 직원용 리포트 생성
```

## 11. TAAS 데이터 역할

TAAS는 개인 사고 예측 학습 데이터가 아니다.

정확한 역할은 아래다.

```text
노인운전자 사고와 법규위반/사고유형 통계를 기반으로
위험행동 가중치와 상품 필요성의 공공 근거를 제공한다.
```

연결은 아래처럼 둔다.

| TAAS 항목 | 우리 피처 |
|---|---|
| 과속 | `speeding_count`, `speeding_per_100km` |
| 안전거리 미확보 | `harsh_brake_count`, `harsh_brake_per_100km` |
| 신호위반/안전운전 의무 불이행 | 위험행동 점수 참고 |
| 차량단독 | 야간 + 급조작 + 생활권 밖 변화 참고 |
| 노인운전자 사고 | 시니어 특약 필요성 근거 |

## 12. XAI + LLM 리포트

XAI는 판정 이유를 구조화한다.

예시 reason code:

```text
low_mileage=true
in_zone_safe_score=high
out_zone_pattern_change=high
outer_night_ratio_increased=true
outer_harsh_brake_increased=true
```

LLM은 이 reason code를 보험사 직원용 문장으로 바꾼다.

예시:

```text
이 고객은 연환산 주행거리는 낮지만, 최근 30일 동안 생활권 밖 주행 비율과 야간주행 비율이 증가했고 해당 구간에서 급감속도 함께 늘었습니다. 따라서 단순 저주행 고객으로만 보기보다 예방 케어 안내가 필요한 고객으로 분류됩니다.
```

중요한 원칙:

```text
LLM은 보험료나 등급을 결정하지 않는다.
LLM은 XAI reason code를 설명문으로 변환하는 역할만 한다.
```

## 13. 절대 하지 않을 주장

아래 표현은 사용하지 않는다.

```text
사고율을 예측했다.
실제 손해율 개선을 증명했다.
TAAS로 개인 사고를 학습했다.
생활권 밖 주행은 위험하다.
생활권 안 주행은 안전하다.
시니어에게 패널티를 준다.
기존 마일리지에 추가 혜택만 붙인다.
```

대신 아래처럼 표현한다.

```text
기존 주행거리 중심 마일리지 산식을
생활권 안정성과 위험변화까지 반영하는 통합 지표로 고도화한다.
```

## 14. 최종 브리핑 문장

```text
본 프로젝트는 기존 마일리지 할인특약을 단순히 확대하는 것이 아니라,
연환산 주행거리, 생활권 내 안정운전, 생활권 밖 안전운전,
생활권 밖 위험변화를 함께 반영하는 시니어 특화 통합 마일리지 산정 모델이다.

출시 전에는 페르소나 기반 배치 시나리오 시뮬레이터로
기존 거리 중심 산식과 제안 통합 산식을 같은 조건에서 비교해
가중치와 케어 기준의 타당성을 검증한다.

실제 운영 단계에서는 고객 주행 데이터에 DBSCAN/P90 기반 생활권 분석을 적용하고,
XAI와 LLM을 통해 보험사 직원이 납득 가능한 판정 리포트를 제공한다.
```
