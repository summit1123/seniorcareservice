# 팀원 공유용 진행상황 정리

## 1. 현재 프로젝트 방향

우리 제안은 **생활권 기반 시니어 안심주행 특약**입니다.

기존 자동차보험 특약은 보통 아래 두 가지 기준을 봅니다.

| 기존 기준 | 의미 |
|---|---|
| 마일리지 특약 | 얼마나 적게 운전했는가 |
| 착한운전 특약 | 과속, 급가속, 급감속 등 안전운전 점수가 좋은가 |

우리 제안은 여기에 세 번째 기준을 추가합니다.

```text
시니어 운전자가 익숙한 생활권 안에서 평소처럼 안정적으로 운전하는가
최근 평소와 다른 위험 변화가 생겼는가
```

즉, 목표는 사고를 직접 예측하는 AI가 아니라 **평소 생활권과 운전패턴을 학습해 추가 리워드 또는 예방 케어로 연결하는 AI 특약 구조**를 보여주는 것입니다.

## 3. AI가 하는 일

| AI 활용 | 설명 | 산출물 |
|---|---|---|
| DBSCAN 생활권 생성 + 버퍼 | baseline 기간의 출발지/도착지를 묶어 반복 생활권 중심을 찾고 P90 버퍼로 주변 이동을 허용합니다 | `zone_feature_table.csv` |
| 평소패턴 이상탐지 | 최근 주행이 평소보다 얼마나 달라졌는지 점수화합니다 | `pattern_change_score.csv` |
| XAI reason code | 왜 추가 리워드/예방 케어인지 주요 이유를 남깁니다 | `decision_table.csv`, `model_demo_summary.md` |

생활권은 단일 좌표가 아니라 아래처럼 나눕니다.

```text
core zone = 반복 생활권 중심 주변 최소 500m
buffer zone = 고객별 baseline 목적지 이탈거리 P90 안쪽
outer zone = 버퍼 밖 외부 주행
```

버퍼 반경은 파일럿 기준으로 아래처럼 계산합니다.

```text
생활권 버퍼 = max(500m, min(P90, 2km))
```

중요한 점은 **outer zone 주행 자체를 위험으로 보지 않는다는 것**입니다.

생활권 밖 주행은 병원, 가족 방문, 장보기처럼 정상적인 이유가 있을 수 있습니다. 따라서 우리 모델은 아래 조건이 함께 나타날 때만 예방 케어 신호로 봅니다.

```text
생활권 밖 주행 증가
+ 과속/급감속/급회전 증가
+ 야간 주행 증가
+ 평소 대비 운전패턴 변화 증가
```

## 4. TAAS 데이터 역할

TAAS 데이터는 개인별 사고를 예측하는 라벨로 쓰지 않습니다.

TAAS는 아래 두 가지 용도로만 사용합니다.

| 사용 방식 | 설명 |
|---|---|
| 상품 필요성 근거 | 고령운전사고, 야간사고, 법규위반 사고통계로 특약 필요성을 설명 |
| 위험행동 가중치 참고 | 과속, 안전거리 미확보, 안전운전 의무 불이행 등 어떤 위험행동을 더 중요하게 볼지 참고 |

따라서 발표나 문서에서는 아래처럼 말해야 합니다.

```text
TAAS는 개인 사고 예측용 데이터가 아니라,
공공 사고통계 기반 위험행동 가중치와 상품 필요성 근거로 활용합니다.
```

## 5. 팀원이 지금 준비하면 되는 데이터

현재 가장 필요한 것은 **사업용차량 주행데이터 CSV 원본**입니다.

원본 컬럼명이 우리 코드와 달라도 괜찮습니다. 매핑 스크립트가 원본 컬럼을 표준 컬럼으로 맞출 수 있는지 먼저 확인합니다.

### 필수에 가까운 컬럼

| 필요 데이터 | 이유 |
|---|---|
| 차량 또는 운전자 식별자 | 같은 운전자의 평소 패턴을 묶기 위해 필요 |
| 운행 시작 시각 | baseline/recent 구분, 야간 여부 계산에 필요 |
| 운행 종료 시각 | 운행시간 계산에 필요 |
| 시작 GPS X/Y | 생활권 출발지 판별에 필요 |
| 종료 GPS X/Y | 생활권 목적지 판별에 필요 |
| 운행거리 | 100km당 위험행동 정규화에 필요 |
| 평균속도 또는 운행시간 | 주행 품질 계산에 필요 |
| 과속건수 | 위험운전 feature |
| 급가속건수 | 위험운전 feature |
| 급감속건수 | 위험운전 feature |
| 급좌회전/급우회전건수 | 급회전 feature |

주의할 점은 **같은 차량 또는 운전자에 대해 여러 Trip이 있어야 한다는 것**입니다.

```text
1명당 1개 Trip만 있으면 생활권 학습과 평소 대비 변화 감지가 어렵습니다.
가능하면 같은 driver_id 기준으로 2개 이상, 권장으로는 3개 이상의 Trip이 필요합니다.
```

`trip_duration_min`은 정수뿐 아니라 `1006.98`처럼 소수 분 단위도 사용할 수 있습니다.

### 있으면 좋은 컬럼

| 데이터 | 쓰임 |
|---|---|
| 최고속도 | 위험운전 보조 지표 |
| 정지건수 | 운전패턴 보조 지표 |
| 누적운행거리 | 데이터 검증용 |
| 운행일자 | 기간 분리 보조 |

## 6. 실제 데이터가 오면 할 일

팀원에게 원본 CSV를 받으면 `data/raw/` 폴더에 넣고 아래 명령을 실행합니다.

```bash
python3 scripts/validate_trip_csv_mapping.py data/raw/<원본파일명>.csv --run-pipeline --generate-visuals
```

이 명령은 아래 작업을 한 번에 수행합니다.

| 단계 | 내용 |
|---|---|
| 1 | 원본 CSV 인코딩과 컬럼 확인 |
| 2 | 우리 표준 Trip schema로 매핑 가능한지 확인 |
| 3 | 필수 컬럼 누락 여부 확인 |
| 4 | 가능하면 표준화된 중간 CSV 생성 |
| 5 | feature table 생성 |
| 6 | 점수와 최종 판단표 생성 |
| 7 | 리포트와 시각화 자료 재생성 |

결과는 주로 아래 파일에서 확인합니다.

```text
.codex-loop/artifacts/csv-mapping/*-mapping-report.md
data/processed/model_feature_table.csv
data/processed/pattern_change_score.csv
data/processed/score_table.csv
data/processed/decision_table.csv
reports/model_demo_summary.md
reports/figures/*.svg
```

## 8. 현재 완성도

현재는 **실제 주행데이터 적용 전의 모델 견본 단계**입니다.

| 영역 | 현재 상태 |
|---|---|
| 아이디어/상품 설계 | 거의 완료 |
| AI 모델 구조 | 완료 |
| 코드 파이프라인 | 샘플 데이터 기준 완료 |
| TAAS 활용 기준 | 완료 |
| 실제 주행데이터 적용 | 대기 중 |
| 발표자료 최종화 | 이후 진행 |

전체 기준으로 보면 현재 완성도는 약 **70~75%** 정도입니다.

실제 주행 CSV가 들어와서 결과표와 리포트를 재생성하면 **80~85%**까지 올라가고, 발표자료까지 정리하면 최종 제출 수준에 가까워집니다.

## 9. 팀원에게 공유할 한 줄

```text
이제 필요한 것은 실제 사업용차량 주행 CSV이며, 원본 컬럼명이 달라도 매핑 검증부터 진행하면 됩니다.
```
